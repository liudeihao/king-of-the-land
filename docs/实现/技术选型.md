# 技术选型（实现向）

> 本文档在 `微内核与Mod扩展.md` 与 `系统结构.md` 的约束下，给出**可执行**的技术栈与数据方案；**不是**对设计假设的重复论证。  
> 与 `原型构建步骤.md` 冲突时：**以本文为准更新原型步骤**（例如已统一为 Unity 则保持），或显式标注「原型专用例外」。（以上与本文同处 `docs/实现/`。）

---

## 0. 结论摘要（TL;DR）

| 领域 | 选型 | 备注 |
|------|------|------|
| 客户端与渲染 | **Unity 2D（URP）** | 与现有《原型构建步骤》一致；2D 俯视、Tile/精灵路线。 |
| 宿主语言 | **C#** | 与 Unity 一致；**模拟核心**建议与 `MonoBehaviour` 渲染层解耦。 |
| 模拟执行 | **固定 Tick + 显式阶段** | 便于测试、存档、后期谈确定性；呈现层与逻辑层分离。 |
| L1 内容 / Mod 数据 | **JSON**（首版）或 JSON + **JSON Schema** 校验 | 可读、易 diff；体积大时可再引入 MessagePack 仅作打包。 |
| 存档 | **版本头 + MessagePack 或 JSON**（二选一，见 §4） | 须含 schema 与 Mod 清单；与 `微内核` §6 一致。 |
| L2 脚本（后期） | **优先 Lua（如 MoonSharp / 绑定审计）**；备选 **C# 受限程序集** | WASM 作远期选项；见 §5。 |
| 随机数 | **可播种、可隔离分流的 PRNG** | 为 `微内核` 2.1 随机性契约服务；**禁止**直接 `UnityEngine.Random` 进模拟核心。 |
| 版本与协作 | **Git**；`*.unitypackage` 仅作外部分发，不作为源码真相 | |
| 单元测试 | **NUnit 或 xUnit** + **可脱离 Unity 的纯 C# 核心程序集** | 关键：模拟不依赖 `UnityEngine`。 |

**当前已确认工程**（以仓库内 `unity/ProjectSettings/ProjectVersion.txt` 为准，2026-04 起锁定记录）：

| 项 | 值 |
|----|-----|
| 编辑器 | **6000.3.13f1**（Unity 6 / 6000.3 线） |
| Hub 模板 | **Universal 2D**（2D + URP；包内含 `com.unity.render-pipelines.universal`） |
| 路径 | 仓库内 **`unity/`** 为项目根（`Assets/`, `Packages/`, `ProjectSettings/`） |

以下各节为理由、边界与可替代方案。

## 1. 游戏引擎与项目形态

### 1.1 主选：Unity 2D

**理由**（与《版本路线图》里程碑二、原型步骤一致）：

- 2D 俯视、生态格网、大量简单个体精灵：Unity 2D + URP 足够，资料与生态成熟。  
- C# 单语言贯通「编辑器工具 / 游戏逻辑 / 将来 Mod 工具链」，团队认知成本低。

**需遵守的架构纪律**（对应 `微内核`）：

- **模拟态**（世界、实体、环境、编年史事件）放在 **Assembly Definition** 划出的 **无 Unity 依赖** 的核心程序集（如 `LandKing.Simulation`）中；仅 **呈现与输入** 引用 `UnityEngine`。  
- 好处：同一核心可被 **命令行/测试宿主** 驱动，便于长时重放、确定性排查（06 #8、#19）。

### 1.2 备选：Godot 4 + GDScript / C#

若团队更熟悉 Godot，可整体迁移：**原则不变**（逻辑核与呈现分离、表驱动数据、固定 Tick）。需单独重写《原型构建步骤》与渲染/资源管线；本阶段若无动机不必分叉。

### 1.3 不优先：自研引擎或纯游戏框架从零搭

**理由**：前文明 sim 的工程量在**内容与模拟**，不在渲染；除非有强定制需求，否则不分散火力。

---

## 2. 语言与核心边界

- **C# 11+**（随 Unity 绑定的 LTS 版本为准）。  
- 命名空间建议：`LandKing.Simulation` / `LandKing.Content` / `LandKing.Mods` / `LandKing.Presentation`（可调整，但须保持**单向依赖**：Presentation → Simulation；Simulation **不**引用 Presentation）。  
- **热路径**（上万个体前）：以清晰为先；若分析确认瓶颈在群体路径，可再引入 **Unity Jobs + Burst** 限定在 **可并行、可声明无托管竞争的子块**（与 Determinism 需求一起评估，见下）。

---

## 3. 模拟调度与时间

- **固定逻辑帧**：例如 `dt` 为游戏日或更小单位，以整数 Tick 计（避免浮点随框架各异导致分歧）。  
- **阶段化更新**（与 `系统结构` 子系统对应）：在单 Tick 内固定顺序，例如：环境 → 生态 → 个体决策 → 移动解析 → 社会/繁殖 → 事件提交 → 编年史。  
- **时间加速**（06 #8）：  
  - **呈现**：每帧可推进多 Tick，或 `while` 批量推进到「下一可绘事件」；  
  - **或** LOD / 子系统降精度：**须在存档中记录**（`微内核` 5.1），且版本化。  
- **暂停**：仅停止推进 Tick，不停止 UI。

---

## 4. 数据、存档与序列化

### 4.1 L1 内容与配置

- **首版推荐 JSON**：表驱动物种、行为参数、事件表、地图生成偏置。  
- **校验**：在加载时做 **JSON Schema** 或手写校验器，失败则拒绝加载并报告 Mod 与字段（`微内核` 2.2）。  
- 若文件过大或需二进制分发：可 **发布管线** 将 JSON 编译为 **MessagePack** 块，**内核仍按同一 schema 反序列化**（玩家侧透明）。

### 4.2 世界存档

- **必须**：`schema_version`、世界种子、`mod_manifest` 哈希/列表、核心 `WorldState` 二进制或结构化块。  
- **选型**：  
  - **MessagePack**（C# 库成熟）：体积与速度好，适合大地图+多实体。  
  - 或 **JSON 压缩/分块**（调试用可读性高）。  
- **要求**：**禁止**在补丁中改字段含义而不升 `schema_version`（`微内核` §6）。

### 4.3 编年史与事件流

- 使用 **只追加** 的事件结构（与 `系统结构` WorldEvent 一致），序列化进存档或分文件（大档时可按「纪元」分卷）。  
- 索引策略（时间线查询）首版可内存构建，再按需加轻量侧车索引。

---

## 5. Mod：L2 脚本与 L3 本机

与 `微内核` §4 对齐：

- **L2 首推 Lua（MoonSharp 等）**：宿主注入**窄接口**（只读世界快照 + 白名单系统回调），**禁止**任意反射与文件系统；超时与步数限制。  
- **备选**：C# 受信 Mod 在**单独程序集**中实现约定接口，由内核 **AssemblyLoadContext** 加载并卸载（实现成本高，但类型安全）。  
- **L3**：Native 插件 = 受信渠道、默认关闭、清单标注 `nondet` 或等效。  
- **WASM**：作为「跨宿主 Mod」的远景；首版不引入，以免绑定与调试成本压垮小团队。  
- **不采用**：「每 Tick 全开放 C# 反射」式 Mod，与可控干预管道冲突。

---

## 6. 网络与联机

- **首版/单人不强制网络**。  
- 若未来有云存档，采用 **独立服务 + HTTPS**，与**模拟核**正交。  
- **不做**在模拟全程依赖服务器权威帧同步（与观察类玩法不符）。

---

## 7. 开发工具与质量

- **IDE**：Rider 或 Visual Studio + Unity 集成。  
- **格式**：`.editorconfig`、统一换行与缩进 C# 常规约定。  
- **测试**：核心程序集上跑 **NUnit**；为 Tick、环境、编年史等写 **金样测试**（固定种子 → 固定若干步 → 比较哈希/摘要，配合 `微内核` 6 附录 A）。  
- **性能分析**：Unity Profiler + Memory Profiler；**先**在逻辑核测瓶颈再 Burst。

## 9. 风险与回滚

- **若 Unity 授权/平台策略不可接受**：整体平迁 Godot 或 Unreal 仅影响 **Presentation** 与资源管线，**要求**已坚持 Simulation 与 Unity 解耦。  
- **若 Mod 需更强逻辑早于 Lua**：先强化 **L1 数据 + 行为组合**，再开 L2，避免未校验脚本污染存档。
