# 微内核与 Mod 扩展

> 本文档规定游戏的**实现架构方向**：以微内核承载不可妥协的模拟契约，以内容与 Mod 承载世界丰富性。  
> 本文是后续工程实现的**结构依据**；未在此承诺具体 API 形态与时间表，但承诺**不违背**下列边界。

---

## 0. 目标

- **内核极小**：只包含使「个体级、可存档、可复现的模拟 + 受约束的玩家接口」成为可能的运行时与数据契约。  
- **外环可替换**：生物种类、行为模板、环境规则、部分 UI 与信息呈现、（受约束的）事件与内容包，以**数据包**或 **Mod** 形式挂接。  
- **与项目愿景对齐**：丰富性不依赖玩家在游玩中「开菜单造内容」，而依赖世界自身运行；Mod 是**开发与世界构建阶段**的扩展面，不替代「观察、注意力、触发式困境」等原则所依托的规格（见 `docs/00_项目宣言.md`、`docs/04_玩家关系层.md`）。

## 2. 微内核：必须包含什么

以下模块属于**内核责任**，其行为构成「同一款游戏」的底线；不随 Mod 替换（若替换则属于「Fork」或主版本不兼容，而非普通 Mod）。

### 2.1 模拟与调度

- **时间推进单位（Tick）的语义**：全局顺序、各子系统读写的阶段划分（可随版本演进，但须在版本间文档化）。  
- **世界状态的最小集**：能支撑存档、读档、（若实现）分支与对比视图的**可序列化状态**与 **schema 版本号**。  
- **实体身份**：稳定 ID 与生命周期规则（出生、死亡、销毁），以及跨子系统引用的安全约定。  
- **随机性契约**：若采用种子，须明确**哪些随机流**影响哪些子系统，以便在「有 Mod 时仍尽可能复现」或明确标注**不可复现**的 Mod 类型。

### 2.2 世界与内容的分界线

- 内核提供 **注册表/类型系统** 的**槽位**（如物种、行为 ID、环境事件类型、干预管道类型），并执行**加载、校验、版本协商**。  
- 具体**填槽内容**由**默认数据包**与 **Mod** 提供；内核不内置「必须有猿与狼」的硬编码业务逻辑，但**可**在「无内容包」时带一份最小可运行内嵌数据（由实现择一，须在版本记录中说明）。

### 2.3 玩家接口的受控入口

- **观察**（读路径）：原则上任意 Mod 可扩展**呈现**（图标、高亮、档案模板），但默认不得伪造世界内未发生的事实。  
- **干预**（写路径，含 MVP 的降雨等）：**只能**经内核登记的**干预管道**进入模拟；禁止 Mod 在通用钩子里**直接**修改环境/个体核心状态而不走管道（见 `04` 干预原则与 `系统结构` 干预章）。新干预类型=新管道定义 + 触发条件 + 代价模型，在文档与数据中可审计。

### 2.4 编年史与事件流

- 世界内发生的事实应能汇总为**结构化事件流**（与 `系统结构` 中 WorldEvent 方向一致），供原生日志、Mod 叙事、回放使用。  
- Mod 可增删**显示层**与**派生统计**，不应改写已提交的历史事实（若允许「重述」，须单独标记为玩家侧解读，而非世界内真值）。

## 3. 外环：默认内容与 Mod

### 3.1 默认内容包

- 随发行或仓库提供的**第一方数据包**（生物、行为、地图生成参数、MVP 场景所需的一切），在语义上等价于「官方 Mod」，仅优先级与签名为默认。  
- 微内核在开发早期即可采用「内嵌**默认表**、尚未对外 Mod 管线」的过渡实现；**工程结构**上仍应按「可替换数据包」拆分，避免业务写死在主程序路径。

### 3.2 Mod 的定义

- **Mod** = 在默认加载管线之外附加的**内容包**与（可选的）**受控逻辑扩展**的集合，带标识、版本、依赖声明。  
- Mod 的丰富度**优先**通过 **数据**（表、资源、行为参数、条件脚本）提供；**通用代码 Mod** 属于高级能力，受 **§4** 安全与确定性约束。

---

## 4. Mod 分级与工程约束

| 级别 | 内容 | 允许做的事 | 必须遵守 |
|------|------|------------|----------|
| **L1 数据 Mod** | JSON/YAML/等价表、贴图/音效、**无**任意代码 | 新物种、新地形与产出规则、在既有行为槽上填参数、新事件表条目 | 通过内核校验；与存档 schema 协同演进 |
| **L2 受控逻辑 Mod** | 沙盒内 DSL / 脚本 / WASM 等（实现待定） | 新决策片段、新条件、新**非破坏**性事件响应 | 禁止无限循环与未声明副作用；**确定性**须在文档中分级标注（`strict` / `best_effort` / `nondet`） |
| **L3 本机 / 主机扩展** | 与宿主同语言的动态库或插件 | 性能敏感或深度扩展 | 仅作为**受信任**通道；默认不开启；不保证存档跨机器一致，须在清单中标明 |

**原则**：L1 应能支撑绝大部分「丰富性」；L2 用于绕不开逻辑的内容；L3 非大众 Mod 的默认路径。

## 5. 与开放问题对应的硬约束

### 5.1 时间加速与模拟（06 #8）

- 内核必须定义：**加速 = 只影响呈现步进，还是允许分级 LOD**。  
- 若某 Mod 或管线注册「简化子系统」，**必须**在存档元数据中记录，以便诊断「同存档、不同 Mod 组合 → 不同世界走向」的因果。

### 5.2 LOD 与可复现性（06 #19）

- 微内核不假装能解决所有算力问题；**允许**在性能模式下降低精度，但：  
  - 要么对玩家不可见地仍保证同一宏观种子下的**可接受**偏差范围（须定义「可接受」的测试方式）；  
  - 要么显式以「非确定性 / 非同步」模式运行并在 UI 中提示。  
- **禁止**在文档未定义的情况下，让 Mod  silent 替换 LOD 策略并污染竞技/分享存档的预期。

### 5.3 干预（06 #16）

- 新干预 = **新管道** + 触发 + 作用域 + 代价；**禁止**仅通过「调数值」实现神迹式干预。  
- 若 Mod 提供新干预，须在依赖清单中声明，并在 UI 上可与原版干预**区分**（避免玩家误认官方叙事）。

### 5.4 节奏、通知、注意力（06 #14、#17）

- **通知与智能加速**的算法可放在**接口层/策略模块**，可做成「可换策略」甚至 Mod；但不得默认用通知**消解**设计目标中的「无法同时看见一切」——该张力由**产品配置**与**Mod 元数据**显式标定。  
- 若 Mod 增加「高提示性」功能，应标注 `spoils_attention_tension: true` 一类元数据，供玩家与实现者做知情选择（字段名为示意，实现时以工程名为准）。

## 6. 存档、版本与兼容性

- 所有持久化块携带 **内容版本** 与 **Mod 集哈希或清单**；打开存档时做**前向/后向**兼容性检查。  
- 破坏性变更走 **主版本** 与迁移工具；**不得**在补丁中悄悄改变 L1 数据语义而不升版本。  
- 「仅数据 Mod」的存档在卸载 Mod 后的行为：要么**拒绝加载**并提示，要么在明确规则下**剥离**该 Mod 内容（以设计决策为准，须在实现前写入本文**修订段落**）。

## 7. 分阶段落地（实现顺序）

1. **阶段 A（与里程碑二～四并行）**  
   - 代码结构按「内核 / 默认数据 / 表现层」分目录；物种与地图参数**表驱动**。  
   - 无对外 Mod 加载器亦可，**但禁止**在核心循环里散落魔法数。  
   - **（原型已部分落地）** `WorldSimulation` 以 `SimParams` 为表、单 Tick 分阶段；Unity 端从 `StreamingAssets/Mods/` 读入 **L1 补丁**（`sim_params.json` 的 `patches` 表）与可选 `mod.json` 元数据，合并后再构造模拟核——属阶段 B 正式清单格式之前的最小试装。  

2. **阶段 B**  
   - 正式 **Mod 清单格式**、加载顺序、冲突解析；**仅 L1** 首亮。  
   - 编年史与事件流稳定后，再接 Mod 派生展示。  

3. **阶段 C**  
   - L2 沙盒、确定性分级、性能预算。  
   - 干预管道插件化。  

4. **阶段 D**  
   - L3 受信任通道（如需）；工坊与签名策略（产品决策）。  

**顺序可压缩，但不可颠倒 2.2「注册表与校验」于大规模 Mod 内容之前。**

---

## 8. 非目标（当前阶段明确不做）

- 不为 Mod 提供「每 Tick 自由 Hook 全宇宙」的通用 API（除非在 L3 中显式承担后果）。  
- 不将**在线多人**纳入内核义务（若未来有，另文）。  
- 不承诺首版即支持跨引擎 Mod 格式。
